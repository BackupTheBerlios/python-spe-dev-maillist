>     You can directly assign to sys.executable.  It's a nice fact about
>     Python that once you have access to a module's namespace you can
>     assign
>     to globals in it. Unfortunately it makes it difficult to optimise
>     Python
>     (because you can't know anything about the type of objects in a module
>     namespace -because they can be overridden at runtime by another module
>     with access to the namespace).
>
>
> I know you can overwrite, but this is only within the namespace of the
> running script. If a script like SPE launches another script
> externally like winpdb, sys.executable won't be changed there. So be
> careful about that.

Scripts are launched by being "exec'd" by movable Python. Changing
``sys.executable`` there affects all programs they run.


>
>     Anyway - running programs with sys.executable works fine. Normally it
>     will be the 'right thing'. In this case it isn't. Hmmm... maybe when
>     launching SPE I can change sys.executable.
>
>
> And also in rpdb2.py. I'm afraid the debugger won't work otherwise.
>
I've hit on an acceptable solution. I'll create a third executable
``movpywc.exe``. It doesn't have a console box (so it is like
``movpyw.exe``) bit it changes sys.executable to point to ``movpy.exe``.
That means that programs that use sys.executable to launch programs will
have a console box as normal.

I can special case SPE to use this - it will also be available to
programmers to use with their programs that (may) launch subprocesses.

It's kind of a hack (special case), but not 'unuseful'. It's certainly
easier than maintaining patches against SPE.
